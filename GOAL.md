### A. Charting (`captain add`)

Setup: `mkdir -p src && echo "export const x = 1" > src/a.ts`

- **A1** — `captain add "Where is X?" "In src/a.ts" --files src/a.ts` exits 0. `.captain/map.jsonl` now has exactly 1 line; it parses as JSON with the question, the answer, and a files array containing path `src/a.ts` (relative, not absolute) and a 64-char hex hash equal to `shasum -a 256 src/a.ts | cut -d' ' -f1`.
- **A2** — Adding with a nonexistent file (`--files src/nope.ts`) exits nonzero and writes no entry.
- **A3** — Re-adding the same question with a different answer leaves exactly 1 entry for it, carrying the new answer (add = delete + add; the map never accumulates duplicates of a question).
- **A4** — From a subdirectory (`cd src && captain add ... --files a.ts`), the stored path is still relative to the project root (`src/a.ts`), and `rg -c '"/(Users|home|tmp|private)' .captain/map.jsonl` finds no absolute paths.

### P. Cache-busting (`captain prune`, `captain delete`)

- **P1** — With files unchanged, `captain prune` exits 0 and deletes nothing.
- **P2** — After `echo change >> src/a.ts`, `captain prune` deletes every entry referencing it; the map no longer contains the question.
- **P3** — Same for a deleted file: `rm src/a.ts` → prune removes its entries.
- **P4** — `captain delete <id>` (id taken from `map.jsonl`) removes exactly that entry.

### R. Recall (`captain context`, `captain list`)

- **R1** — With a fresh entry present, `captain context` stdout contains the question, the answer, and a charting contract that tells the agent to run `captain add` when it discovers something.
- **R2** — `captain context` prunes first: after modifying a referenced file, its entry does not appear in the output.
- **R3** — With an empty/absent map, `captain context` still prints the charting contract and exits 0 (a blank chart must not break SessionStart).
- **R4** — `captain list` prints question, answer, id, and file paths for each entry.
